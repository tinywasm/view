# PLAN — Agregar `ModelSlice` a `tinywasm/model`

> Este plan se ejecuta en el repositorio `tinywasm/model` (mover este archivo allí).
> Es **prerrequisito** del refactor de `tinywasm/view` (`docs/PLAN.md` en ese repo):
> una vez publicada la versión de `model` con este cambio, se ejecuta el plan de view.

## Por qué (el defecto está aquí, no en el consumidor)

`view.Presenter.Reload()` recibe una lista que debe cumplir DOS contratos a la vez:

- `model.FielderSlice` — iterarla sin reflexión (`Len`/`At`/`Append`).
- `model.Decodable` — que el transporte (`router.Caller`) decodifique la respuesta en ella.

Hoy esa combinación no tiene nombre en `model`, así que `view` la verifica en runtime:

```go
// view/presenter.go (estado actual — el síntoma)
dec, ok := list.(model.Decodable)
if !ok {
	return fmt.Err("view: list returned by newList does not implement model.Decodable")
}
```

El propio doc de `model.Model` declara la regla:

> "Consumers MUST NOT declare local intersections of the atoms below; if a boundary
> needs a combination that this package does not name, **that is a defect HERE**,
> not in the consumer."

La frontera existe (view ↔ transporte), la combinación existe, el nombre falta.
Nombrarla aquí convierte un check de runtime en un error de compilación
(principio 6 del arnés: compile error > loud diagnostic > never silent).

## Cambio

En `interface.go`, junto a `Model` (es su espejo del lado lista):

```go
// ModelSlice is the list-side mirror of Model: a typed slice of records that can be
// iterated without reflection (FielderSlice) and travel from the wire (Decodable).
//
// It is THE type to name at any boundary that receives a decoded list of records —
// a view reloading its items, a Caller filling a response. Consumers MUST NOT
// declare this intersection locally.
//
// Implementations are generated by ormc — never written by hand.
type ModelSlice interface {
	FielderSlice
	Decodable
}
```

Notas de diseño:

- **Solo lectura de red**: NO incluye `Encodable`. La frontera que motiva el tipo es
  "lista que llega del wire"; si más adelante una frontera real necesita enviar listas,
  se nombra esa combinación en su momento (superficie mínima, principio 5).
- **Sin cambio de comportamiento**: es una interfaz nueva; ningún código existente se
  rompe. Las listas generadas por `ormc` ya satisfacen ambos átomos, así que la
  implementan automáticamente sin regenerar nada.

## Test con forma de consumidor (obligatorio antes de publicar)

Regla del arnés: "An API is not published until a consumer-shaped test, inside the
library itself, proves it." Agregar al test suite de `model` una prueba que haga lo que
hará `view`: recibir un `ModelSlice`, decodificar en él y luego iterarlo — con una lista
generada real (la que ya usen los tests de `model`/`ormc`):

```go
func TestModelSliceNamesTheListBoundary(t *testing.T) {
	// El uso con forma de consumidor: una función que exige el contrato combinado,
	// como lo hará view.New(..., newList func() model.ModelSlice, ...).
	var newList func() model.ModelSlice = func() model.ModelSlice { return &TestItemList{} }

	list := newList()
	// 1. Llega del wire (lado Decodable) — usar el codec real de los tests del repo.
	decodeInto(t, list, sampleWirePayload)
	// 2. Se itera sin reflexión (lado FielderSlice).
	if list.Len() == 0 {
		t.Fatal("decoded list must be iterable")
	}
	_ = list.At(0).Schema()
}
```

(Adaptar nombres al modelo de prueba y codec que el repo ya use; lo invariante es que
la MISMA variable ejerza los dos lados del contrato, probando que la intersección
nombrada es utilizable tal cual por un consumidor.)

## Ormc (verificación, no cambio)

`ormc` no requiere cambios: las listas que genera ya implementan `FielderSlice` y
`Decodable` por separado. Verificar con una aserción de compilación en el código de
test de `model`:

```go
var _ model.ModelSlice = (*TestItemList)(nil)
```
